Arquitetura ImobSmart MVP — 3 Abordagens

Qual stack e arquitetura para o MVP? Cada opção considera sua VPS com EasyPanel + n8n

A

🏆 Next.js Monolito + n8n (Recomendado)

Frontend + Backend: Next.js (App Router) com Supabase para DB/Auth

Automações: n8n para autopostagem (IG/FB/TikTok), agente IA WhatsApp, workflows

Deploy: 2 containers no EasyPanel (Next.js + n8n, Supabase cloud free tier)

Prós

  • Stack que vocês já dominam (NK, ClaveHogar usam Next.js)
  • n8n já está na VPS — reusa toda a infra de automação
  • Supabase free tier = DB + Auth + Realtime grátis
  • Mais rápido de construir (1 app, 1 deploy)

Contras

  • Escala limitada no início (ok para MVP)
  • n8n tem limites em execuções simultâneas
B

Next.js + API separada (FastAPI/Python)

Frontend: Next.js

Backend API: FastAPI (Python) para lógica de negócio, IA, scraping

Automações: n8n ou Celery workers

Deploy: 3+ containers no EasyPanel

Prós

  • Python excelente para IA/scraping (o script do filho pode integrar direto)
  • Separação clara frontend/backend
  • Mais flexível para escalar cada parte

Contras

  • 2 linguagens para manter (JS + Python)
  • Mais complexo de deployar e debugar
  • Mais lento de construir
C

Low-code: n8n como backend principal

Frontend: Next.js (só UI)

Backend: n8n faz tudo — API, automações, IA, CRM

DB: Supabase ou Postgres direto

Prós

  • Mais rápido de prototipar
  • Visual, fácil de modificar workflows
  • Menos código para manter

Contras

  • n8n não foi feito para ser API principal — frágil sob carga
  • Difícil debugar quando cresce
  • Vendor lock-in no n8n
  • Limitações de performance sérias
